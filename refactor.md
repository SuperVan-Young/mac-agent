# Refactor Agent 执行规范

本文档定义 `Refactor Agent` 的核心任务：观察并重构 `rtl/generate.py`，把当前实现整理成更容易分析、扩展和继续优化的形态。

## 1. 核心目标

`Refactor Agent` 的重点不是做一轮局部 QoR 调参，而是做代码重构。重构应尽量服务以下目标：

1. 把当前某些硬编码策略，抽象成更通用机制下的一个具体策略实现。
2. 让模块边界更清晰，使后续 agent 更容易分析、替换和新增优化。
3. 在可能的情况下，为后续工作留下更好的结构基础，例如：
   - 更清晰的 IR，便于施加更多优化变化
   - 更清晰的 cost model，便于评估优化价值
   - 更清晰的策略接口，便于替换或组合不同优化策略
4. 尽量保持代码性能不退，严格保证正确性不变。

## 2. 工作边界

- 主要工作对象是 `rtl/generate.py`。
- 不要改评估框架，不要扩大任务边界，不要为了重构顺手做大规模优化试错。

## 3. 对重构的要求

- 先理解现有 `generate.py` 的结构和当前策略，再动手重构。
- 重构的重点是理清边界、显式化机制、减少耦合，而不是只做重命名或格式整理。
- 如果当前实现里有一些策略、规则、启发式是写死在流程里的，应尽量把它们变成更清楚的抽象。
- 重构后应尽量保持行为一致；如果行为有变化，必须能解释原因。

## 4. 关于优化

如果重构完成后，存在非常明显的 low-hanging fruit，且满足以下条件，可以立即一起做掉：

- 改动很小
- 逻辑很清楚
- 风险很低
- 预期收益显著

否则，不要在本轮继续扩张；把值得尝试的优化方向写入 `promote.json`，留给后续 agent。

## 5. 输出要求

完成本轮后，应能清楚说明：

- 当前识别到的主要耦合点或硬编码点是什么
- 本轮抽象出了什么机制
- 当前保留的具体策略实现是什么
- 是否发现了值得后续推进的优化方向
- 重构后正确性和性能是否保持稳定
